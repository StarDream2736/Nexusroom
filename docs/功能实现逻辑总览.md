# NexusRoom 功能实现逻辑总览

> 依据 `docs/NexusRoom.md`（v1.3.2）与当前仓库代码整理。
> 本文聚焦“实现逻辑与代码映射”，用于研发/测试/运维快速定位。

---

## 1. 系统启动与依赖装配

### 1.1 服务端启动链路
- 入口：`server/cmd/server/main.go`
- 核心流程：
  1. 读取 `config.yaml`
  2. 初始化 PostgreSQL、自动迁移
  3. 初始化 Redis（健康探测）
  4. 组装 Repository（user/room/message/ingress/friend/wgPeer）
  5. 启动 WebSocket Hub
  6. 注册 Gin Router
  7. 启动消息清理定时任务（retention）
  8. 启动 HTTP 服务并处理优雅关闭

### 1.2 客户端启动链路
- 入口：`client/lib/main.dart`
- 核心流程：
  1. Flutter 绑定初始化
  2. 初始化窗口管理（最小尺寸/标题）
  3. ProviderScope 启动
  4. 读取 `appRouterProvider` 决定路由守卫与跳转
  5. 初始化窗口生命周期服务（后台资源管理）

## 2. 认证与用户模块

### 2.1 注册/登录
- 服务端入口：
  - `POST /api/v1/auth/register`
  - `POST /api/v1/auth/login`
  - 文件：`server/internal/api/handler/auth.go`
- 逻辑：
  - 注册：校验参数 -> 查重 -> bcrypt -> 角色判定（admin_token）-> 创建用户 -> 生成 JWT
  - 登录：查用户 -> 验证密码 -> 检查 `is_active` -> 更新时间 -> 返回 JWT

### 2.2 用户资料
- 服务端入口：
  - `GET /api/v1/users/me`
  - `PATCH /api/v1/users/me`
  - `POST /api/v1/users/me/avatar`
  - `GET /api/v1/users/search?display_id=...`
- 关键文件：`server/internal/api/handler/user.go`
- 头像上传逻辑：
  - multipart 上传
  - 大小限制（来自 `storage.max_file_size_mb`）
  - Header Content-Type + magic bytes 双重校验
  - 保存到 `uploads/avatars`，返回相对 URL

### 2.3 客户端调用
- 用户仓库：`client/lib/features/user/data/user_repository.dart`
- 设置页：`client/lib/features/auth/presentation/pages/settings_page.dart`
- 关键点：设置页头像更新已走 `/users/me/avatar`，不再复用房间文件上传。

## 3. 房间与成员管理

### 3.1 房间管理 API
- 服务端入口：`server/internal/api/handler/room.go`
  - `POST /rooms` 创建
  - `POST /rooms/join` 加入
  - `GET /rooms` 列表
  - `GET /rooms/:roomId` 详情
  - `PATCH /rooms/:roomId` 修改
  - `DELETE /rooms/:roomId/members/:userId` 踢人

### 3.2 创建房间流程
1. 校验请求
2. 创建房间记录
3. 创建者自动入房（admin）
4. 尝试创建默认 Ingress
5. 返回房间信息 + `ingresses`

### 3.3 删除房间（超管）
- 文件：`server/internal/repository/room.go`
- 逻辑：事务级联删除
  - room_members
  - messages
  - room_ingresses
  - wg_peers
  - rooms

## 4. 消息系统（HTTP + WebSocket）

### 4.1 历史消息拉取（HTTP）
- 服务端入口：`GET /rooms/:roomId/messages`
- 文件：`server/internal/api/handler/message.go`
- 策略：
  - `after_id`：增量同步（向后）
  - `before_id`：加载更早消息（向前）
  - 默认：最新 N 条

### 4.2 客户端本地缓存与同步
- 本地库：drift（`client/lib/core/db`）
- 同步入口：`client/lib/features/room/data/message_repository.dart`
- 逻辑：递归/循环拉取直到返回条数 `< limit`
- UI 消费：`messagesStreamProvider` 订阅本地数据库

### 4.3 WebSocket 实时消息
- 服务端 Hub：`server/internal/ws/hub.go`
- 客户端连接：`client/lib/core/network/ws_service.dart`
- 协议：Envelope（event/room_id/payload/timestamp-UTC）
- 事件：chat.send -> server 持久化 -> 广播 chat.message -> 客户端写入本地 DB

---

## 5. WebSocket 在线事件与状态

### 5.1 连接与心跳
- 连接成功服务端发 `connected`
- 客户端收到后启动 30s 心跳
- 断线重连采用指数退避（1/2/4/8/16/30s）

### 5.2 成员与语音状态
- 事件：
  - `room.member_join`
  - `room.member_leave`
  - `voice.state_update`
  - `room.kicked`
- 客户端房间页（`room_detail_page.dart`）已订阅并触发房间详情刷新（踢出场景会导航回首页）

### 5.3 并发安全
- Hub unregister 使用指针身份检查，避免重连竞争误删新连接
- Client send channel 使用 `sync.Once` 安全关闭，避免 double-close panic

## 7. 文件模块（图片/文件上传）

### 7.1 上传接口
- 入口：`POST /api/v1/files/upload`
- 文件：`server/internal/api/handler/file.go`
- 逻辑：
  1. 必须传 `room_id`
  2. 校验上传者是该房间成员
  3. 校验大小、扩展名、magic bytes
  4. 保存至 `storage/private_files`
  5. 返回 `file_id/url/mime_type/size_bytes/file_name`

### 7.2 文件访问接口
- 入口：`GET /api/v1/files/:fileId`
- 逻辑：
  - 从 `file_id` 解析 room_id
  - 校验访问者是房间成员
  - 返回文件流

### 7.3 聊天图片发送流程
- 客户端先调用 upload（带 room_id）
- 再发 `chat.send(type=image, content=url, meta=...)`
- `meta` 包含 `file_id/file_name/mime_type/size_bytes`

## 8. VLAN（WireGuard）

### 8.1 API
- `POST /rooms/:roomId/vlan/join`
- `DELETE /rooms/:roomId/vlan/leave`
- `GET /rooms/:roomId/vlan/peers`
- 文件：`server/internal/api/handler/vlan.go`

### 8.2 服务端协调
- `server/internal/wg/coordinator.go`
- 逻辑：
  - 房间内分配唯一虚拟 IP
  - 注册/注销 peer
  - 返回服务端 WG 配置
  - 广播 `vlan.peer_update`

### 8.3 客户端 VLAN 面板
- `client/lib/features/vlan/presentation/widgets/vlan_panel.dart`
- 逻辑：
  - 生成密钥对
  - join 获取配置
  - `WireGuardService.startTunnel(WgConfig)` 启动隧道
  - 监听 `vlan.peer_update` 自动刷新成员

## 12. 与规范对照结论（当前代码）

### 12.1 已对齐关键项
- WebSocket Envelope `room_id` 顶层字段
- 30 秒心跳 + connected 后启动
- 消息增量同步使用 `id` 锚点
- 文件上传两步闭环（先 HTTP 上传，再 WS 发送）
- VLAN Join 返回可用 server public key（由私钥推导）
- 房间删除事务级联清理

### 12.2 当前仍建议继续优化项
1. 客户端图片显示当前为直接 `Image.network`，未附带鉴权头；若后续改为严格私有下载，需要统一图片加载组件。
2. 文件访问 `file_id` 采用编码解析 room_id 的无状态方案，后续可升级为数据库元数据表，支持审计与回收策略。
3. `admin/config` 当前仅持久化了 `message.retention_days`，后续可扩展为白名单字段批量持久化。
